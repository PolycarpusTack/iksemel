Three approaches to Electron + Database integration

Pick the architecture before we design the details

A

electron-vite + Context Bridge — Recommended

Use electron-vite as the build tool (minimal changes to existing Vite config). Main process manages all DB connections. A context bridge exposes a typed window.electronAPI to the renderer — the only way the renderer talks to Node.js. electron-builder handles packaging, installers, and auto-updates.

Renderer (React) → window.electronAPI → Preload (context bridge) → IPC → Main process → pg / oracledb

Pros

  • Industry standard Electron pattern
  • Secure — renderer never touches Node directly
  • Hot reload works in dev
  • Existing Vite config mostly unchanged
  • electron-builder handles Windows + macOS packaging

Cons

  • Adds electron-vite as a new build dependency
  • Oracle Instant Client must be installed separately
B

Local HTTP sidecar

Main process spawns a local Express/Fastify server on a random port. Renderer calls http://localhost:PORT for all DB operations — essentially treating the main process as a mini backend. Packaging via electron-builder.

Renderer (React) → fetch(localhost:PORT) → Express → pg / oracledb

Pros

  • Renderer code is familiar REST-style
  • Could reuse same backend for a future web version

Cons

  • Open local port is a security surface
  • Port conflicts possible
  • Extra HTTP overhead vs direct IPC
  • Unnecessary complexity for a desktop app
C

Electron Forge + Vite plugin

Use Electron Forge for the full lifecycle: scaffolding, packaging, publishing, and auto-updates. Vite is added as a plugin. More opinionated structure than electron-vite but more batteries-included for distribution workflows.

Renderer (React) → context bridge → IPC → Main process → pg / oracledb
+ Forge handles publishing to GitHub Releases, Squirrel auto-update, etc.

Pros

  • Best-in-class distribution tooling
  • Built-in auto-update via Squirrel
  • GitHub Releases publishing out of the box

Cons

  • Larger migration — Forge wants to own the project structure
  • Steeper learning curve
  • Overkill if distribution is manual/internal for now
